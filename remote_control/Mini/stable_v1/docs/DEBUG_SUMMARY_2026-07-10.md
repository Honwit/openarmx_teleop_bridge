# OpenArm Mini -> OpenArmX 双臂遥操调试问题汇总（2026-07-10）

## 目标

使用已标定的 OpenArm Mini（双手）遥控基于灵足时代电机的 OpenArmX 双臂，并保证稳定与安全。

## 今天遇到的问题与结论

### 1) 标定命令参数不匹配
- 现象：标定阶段命令报参数错误。
- 根因：设备类型和串口参数名称使用了不匹配版本。
- 处理：统一按当前可用设备类型与参数形式执行。
- 经验：先用 `--help` 或文档确认参数名，不混用历史命令。

### 2) 串口权限问题（`dialout`）
- 现象：无法访问 `/dev/ttyACM*`。
- 根因：用户权限不足。
- 处理：加入串口权限组后重新登录生效。
- 经验：先验证 `ls -l /dev/ttyACM*` 和用户组权限。

### 3) 电机电压/上电异常
- 现象：驱动报低压或上电异常。
- 根因：硬件供电不稳定或未按顺序上电。
- 处理：检查电源、电缆、上电顺序。
- 经验：先排除硬件侧，再排查软件侧。

### 4) 依赖缺失与环境冲突
- 现象：`ModuleNotFoundError: lerobot`，或 ROS2 节点环境不一致。
- 根因：ROS2 Humble（系统 Python 3.10）与 LeRobot（venv Python 3.12）混用。
- 处理：采用“UDP 解耦”架构。
  - ROS2 桥接节点在系统环境运行。
  - MINI 读数发送节点在 `.venv` 运行。
- 经验：不同 Python 生态分离运行，不直接互相 import。

### 5) CAN 接口映射错误
- 现象：`can2/can3` 启动失败，`/tf` 不发布或跟随不动作。
- 根因：当前机器有效接口是 `can0/can1`。
- 处理：双臂跟随固定改为 `right=can0`、`left=can1`。
- 经验：先用 `ip link` 核对接口，再启动 bringup。

### 6) 旧 launch 参数误用
- 现象：用旧参数启动单臂桥接失败（如 `leader_right_can` 等）。
- 根因：架构已改为 UDP 输入，不再由桥节点直接读 CAN。
- 处理：使用新的 `mini_to_openarmx_bimanual.launch.py`。
- 经验：桥节点参数要与当前源码一致，避免复用过时命令。

### 7) 运行中突跳、抖动与硬限位碰撞
- 现象：突然动作过快、末端抖动、关节撞限位并伴随异常日志。
- 根因：输入噪声 + 速度过高 + 软限位不够保守。
- 处理：在 `follower_bridge_node.py` 增强安全机制。
  - 角度/夹爪单位转换。
  - 分左右臂软限位（含肩关节内收更严限制）。
  - 每周期步长限制（限速）。
  - deadband + smoothing 抑制抖动。
- 经验：默认先“保守参数”，确认不碰撞后再提速。

### 8) ROS 参数类型错误导致节点崩溃
- 现象：参数声明报类型不匹配（空列表被识别为 BYTE_ARRAY）。
- 根因：数组参数默认值声明方式不正确。
- 处理：修正参数声明与读取逻辑，保证浮点数组类型一致。
- 经验：数组参数不要用无类型空值，优先显式类型。

## 最终稳定运行形态

- Follower：`openarmx_bringup` 双臂，`can0/can1`。
- Bridge：`mini_to_openarmx_bimanual.launch.py`。
- Sender：`.venv` 中运行 `mini_udp_sender_bimanual.py`。
- 数据链路：MINI -> UDP -> ROS2 Leader Topic -> Follower Bridge -> Position Controller。

## 当天验证结果

- 右臂遥操可用。
- 左臂遥操可用。
- 双臂同时可用。
- 调低速度后，慢速放下不再碰撞。

## 后续建议（换设备时优先检查）

1. 串口号是否变化：`/dev/ttyACM*`。
2. CAN 名称是否变化：`can0/can1` 不一定固定。
3. 关节零位与软限位是否匹配新机械结构。
4. 初始步长和滤波参数先用保守值。
5. 先单臂验证，再双臂联调。
